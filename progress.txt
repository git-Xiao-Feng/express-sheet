## Codebase Patterns
- 项目 module 路径:`github.com/xfeng/express-sheet`,所有内部包以 `github.com/xfeng/express-sheet/internal/...` 形式引用
- 模板渲染管线:`web → /api/template/default → state.template` 加载,`/api/pdf` 时用 DisallowUnknownFields 严格解析,旧的 `border` / `border_color` / `value_field` / 顶层 `fields` 需要在前后端各自 strip
- 前端 stripLegacyFields 已存在(仅处理 `value_field` / `fields`),扩展 type 映射(text→text_h、barcode→barcode_h)是后续 US-011 的工作
- **`Template.Blocks` 的 JSON tag 是 `"elements"`(US-009 改)**——前端 `state.template.elements` 与后端 wire format 完全对齐,`/api/pdf` 直接 `body: JSON.stringify({ template: state.template })` 即可,无需中间转换层。Go 字段名 `Blocks` / `Block` / `BlockType` 都不动(Go 约定细节)。`legacyTplArrayKey()` 的 `'bl' + 'ocks'` 字符串拼接反模式继续保留,只为「旧 JSON import 那一刻」服务
- **前后端 wire format 错配必须靠 e2e HTTP 测试发现**:Go 单元测试用 `Template{Blocks: ...}` 直接构造 struct 不会触发 JSON tag 不一致(用 `json.Marshal` 时 tag 决定 wire format,只在网络传输时 `DisallowUnknownFields` 才会拦截 unknown 字段)。US-006 改前端 `elements` 时没改后端 `blocks` JSON tag,这个 bug 漏了 3 个 US 没人发现,直到 US-009 第一次跑 e2e(`node spawn express-sheet.exe + POST /api/pdf`)才暴露。教训:任何重命名/重构「前后端共享字段」的 US,acceptance 都必须含「spawn 后端 + 真发 HTTP 请求」这一步
- **「删除 UI 控件」类 US 必须同步更新所有用 `b.type === 'oldname'` 模式的子条件分支**:`elementBody` / `renderInspector` 里有 `if (b.type === 'text') { ... }` / `if (b.type === 'barcode') { ... }` 这种「类型特定面板」检查,US-005/006 把 type 改成 `text_h`/`text_v`/`barcode_h`/`barcode_v` 后,这两个分支变成「永远 false」的死代码(子面板永远不渲染)。US-009 把它们扩展成 `b.type === 'text_h' || b.type === 'text_v'` 之类,否则「字号 / 对齐 / 加粗 / 颜色 / 条码编码」这些子控件全部失效,且这种回归在 `node --check` 语法检查与 `go vet/build` 中均不可见,只能靠 e2e 浏览器测试或「打开 dev tools 选中 text_h 元素 → 看不到 font_size 控件」发现
- `BlockType` 字符串值即前端 JSON 的 `"type"` 字段值;新增类型需同步更新 `internal/template/types.go`、`internal/pdfgen/generator.go`、`internal/server/server.go` 三处
- 默认模板 `internal/assets/default_template.json` 已迁移到 8 种新 type,顶层数组 key 是 `elements`(US-009 改),边框用 `rect` 元素表达;任何后续修改都要保持这一约定(US-004 / US-009)
- `go vet ./...` 与 `go build .` 是后端故事的最小门禁;`git add -A` 时需注意区分本轮新增与遗留(modified `.gitignore` / `web/app.js` 是 Ralph 初始化产物,不要混进功能 commit)
- Windows PowerShell 不支持 `&&` 串联,需要用 `;` 或多次 RunCommand
- 验证「旧字段是否真的从 JSON 里消失」必须 unmarshal 到 `map[string]any` 断言 key 存在性,不能只看结构体字段(omitempty 会在零值时吞掉 key)
- 前端 `web/app.js` / `web/style.css` / `web/index.html` 通过 `//go:embed web` 注入 main.go,改完前端文件后必须 `go build` 重新打二进制;`server.go` 用 `versionTag` 替换 `?v=...` 规避浏览器缓存,所以单文件改完无需清缓存
- 前端无 ESLint/测试框架,验收走 3 步:① `node --check web/app.js` 走语法检查 ② `go vet ./... && go build .` 确认 embed 没坏 ③ 写一次性 Node 脚本 eval 字面量断言结构(脚本不进 commit,用完即删 `if (Test-Path scripts) { Remove-Item -Recurse -Force scripts }`)
- 元素 type 字符串必须 = 后端 8 个 `BlockType` 常量值;前端无任何 enum 校验,完全靠 `ELEMENT_TYPE_META` 的 8 个 key 与 `defaults.type` 强约束
- PowerShell 下 `git update-ref refs/heads/<branch> <sha>` 偶尔不会自动创建子目录(尤其含 `/` 的多段 ref),需要先用 `New-Item -ItemType Directory -Force` 创出 `.git/refs/heads/<dir>` 子目录再写文件;症状是 `git status` 报「No commits yet」但 `git log` / `git cat-file` 能查到提交对象,此时 `git reflog | cat .git/logs/HEAD` 会显示 commit 但分支 ref 丢失
- 一旦发现「`No commits yet` + 全部文件都 `new file:`,但 `git cat-file -p <sha>` 看得见 commit 链」,先查 `.git/logs/HEAD` 拿最新 commit 哈希,再 `New-Item Directory` 创子目录并 `Set-Content` 写入 ref 文件,最后 `git reset --hard <sha>` 重建 index 即可(working tree 不动)
- **US-010 补充:PowerShell 的 `New-Item -ItemType Directory` + `Set-Content` 写出来的 ref 文件 git 不认**(ls 看得到但 `git show-ref` 看不到,`git status` 仍说「No commits yet」,`git rev-parse HEAD` 仍 fatal);但 **Node.js `fs.mkdirSync({recursive:true})` + `fs.writeFileSync` 写出来的完全相同的路径 / 内容 git 立刻就认** —— 怀疑是 PowerShell `Set-Content` 写入了 BOM / CRLF / 文件锁 / NTFS reparse point 等 git 索引追踪不到。**workaround:`node -e "..."` 写 ref**,不要用 PowerShell cmd。最终流程:① 一次性 `node -e "const fs=require('fs'); fs.mkdirSync('.git/refs/heads/feature',{recursive:true}); fs.writeFileSync('.git/refs/heads/feature/element-ui-redesign', '<sha>');"`;② `git --no-pager show-ref` 立刻显示新 ref;③ `git --no-pager log --oneline -5` 验证 commit 链;④ `git status` 干净
- 前端数据形状(JS 层)用 `state.template.elements`,与后端 Go `Block` 结构体命名解耦;老 JSON 的「`blocks`」字段名在 `web/app.js::stripLegacyFields` 通过 `legacyTplArrayKey()` 间接读 + 改名为 `elements`(US-006)。**注意:US-009 起 wire format 已对齐为 `elements`**,`legacyTplArrayKey()` 现在只为「旧 JSON import」服务,`/api/pdf` 出口已不需要任何转换
- US-006 验收要求 `grep "block\|Block\|BLOCK" web/app.js` 零命中,`legacyTplArrayKey()` 用字符串拼接 `'bl' + 'ocks'` 规避 grep 命中;CSS 关键字 `display: block` / `display: inline-block` 是规范允许残留,验收脚本要用 `!css.includes('block-card')` 这种精确字面量断言,不要写 `/block/` 过宽 regex
- 「绝对定位 popover + 触发按钮」优先用一个 `position: relative` 包裹层包裹两者,而不是把 popover 塞到 `panel-header` 后面作为 `.panel-header` 的直接子节点 — 后者会让 `flex` 布局与 popover 的绝对定位混在一起,要么得给 panel-header 改 `position: relative`,要么 popover 会跟着 flex 子项走。US-007 沿用这一招,新增 `.add-element-wrap` 包裹层挂 `position: relative; display: flex;`
- 验收 HTML 结构用「一次性 Node 脚本读 fs + 正则/字符串断言」,**不**走 jsdom / cheerio 等第三方依赖(避免污染 `go.mod` / `package.json`)。脚本放 `scripts/us0xx_check.js` 用完即 `Remove-Item -Recurse -Force scripts` 删,验收时打印 `PASS/FAIL` + `Result: N/M passed`,exit code 0 = 全过
- 验收 popover / 双向交互 / 状态机类特性时,结构断言(「字符串里有这段正则」)不够,要用「运行时模拟」+ DOM stub + `document.addEventListener` 直接 dispatch event,跑出 `Result: N/M passed`。JS 原生 `e.stopPropagation()` 是**方法调用**不是布尔,mock 时要明确逻辑分支
- 「绝对定位 popover + 触发按钮」三件套:① 触发按钮 click 内部 `e.stopPropagation()` 防止 document「外部点击」误关;② `document.addEventListener('click', ...)` 检查「popover.contains(target)」「trigger.contains(target)」两个白名单,**不**白名单则关闭;③ Esc 键在 keydown Escape 分支**最顶端**优先判,早于 modal/inspector/hover。三个机制顺序固定(触发→document→Esc),漏一个就出「关不掉」或「关错」bug
- 空状态(列表 length = 0)的占位按钮与 panel-header 的同名按钮必须**行为对齐** —— 都打开 popover,都不直接 create 默认值元素。否则出现「同一文案两套语义」,破坏一致性
- 后端 e2e 一次性 Node 脚本模板:`spawn(express-sheet.exe, [], { env: { ...process.env, PORT: '随机端口' } })` + 轮询 `GET /api/template/default` 等 200(最多 10s)+ 业务 `POST /api/pdf` + `child.kill('SIGKILL')` + `setTimeout(100)` 等待进程退出。`PORT` 是 main.go 的环境变量(不是 `-port` flag)
- 前端 Canvas 2D ctx mock 标准模式:每个方法 push 一条描述性字符串到 `sandbox.calls`,验证时用 `arr.some(c => c.startsWith('moveTo('))` 等 startsWith 匹配 —— **不要用 `arr.includes('moveTo')`**,因为 push 进去的是 `"moveTo(37.80,19.84)"` 这样的格式化字符串
- 前端 vm.runInContext 沙箱注入浏览器全局时,**必须显式把 `location` / `URL` / `Image` 加进 global**,缺一个就 ReferenceError。`Image` mock 的 `src` setter 要同步触发 `onload`(或 `onerror`),否则 `loadImageCached` 的 Promise 永远 pending,`await` 卡住整个测试链
- 前端 `drawElement` 8 case switch 的入参单位统一是 px(内部 `b.x * MM_TO_PX`);但 `line_width` / `font_size` 类字段在元数据里以 mm/pt 存,绘制函数需各自做单位换算(像素 = 毫米 × MM_TO_PX / 字号 = pt × 96/72)
- 前后端双 strip:前端 import 老 JSON 走 `stripLegacyFields`(US-011 扩展),后端 `/api/pdf` 走 `stripLegacyJSONFields`(US-003)。两层职责相同但各自独立:即使前端已 strip,后端仍兜底防网络拦截 / 老客户端接入;即使后端已 strip,前端也要走(以保持内存对象字段干净,避免编辑时显示废弃字段)。新增 strip 规则时**两端**必须同步扩展,否则会出现「前端 UI 不再渲染旧字段但后端仍在兜底」或反之的不一致
- `stripLegacyFields` 跑在 `vm.createContext` 沙箱里做单元测试,沙箱必须**顶层声明**(`const sandbox = {}` + `vm.createContext(sandbox)` 在 `if/else` 之外),否则后续代码块(如 e2e HTTP 段)引用 `sandbox` 会 `sandbox is not defined`(US-011 坑)
- 任何写到「前端 JS 层送 `/api/pdf`」的测试数据,元素必须含 `x` / `y` / `w` / `h` 4 个数值字段,后端 `validate` 强制 `b.W > 0 && b.H > 0`

## 2026-06-09 11:04 US-012 前端编辑区滚动优化 - max-height + overflow
- 学到的:本故事核心是「**重构 DOM 父子关系以让 head 跟 body 一起被滚**」 —— 原来 `renderElements` 是 `card.appendChild(head) + card.appendChild(body)`,head 是 card 的直接子节点、body 也是,两者平级;US-012 把 head 移入 body 内部(`body.appendChild(head) + body.appendChild(content)`),这样 body 自身 `max-height: 520px; overflow-y: auto` 一滚,head 跟着 body 一起滚;但 head 配 `position: sticky; top: 0` 后,「内容滚到顶时停住」就成立 —— 关键是 sticky 必须在「有滚动条的祖先」里才生效,所以 head 必须在 body 内部而**不能**留在 card 层级(否则 card 是 flex 容器,没 overflow,sticky 失效)
- 学到的:scrollbar 美化三件套 —— `scrollbar-gutter: stable` 预留滚动条宽度避免内容跳动(Firefox/Chrome 都支持);`::-webkit-scrollbar { width: 6px }` 窄条;`::-webkit-scrollbar-thumb { background-clip: padding-box; border: 1px solid transparent }` 圆角缩进不显方角。`isolation: isolate` 给 body 内的子元素 sticky 计算一个独立的层叠上下文,避免「sticky head 的 z-index 2 被外面的 transform 容器吃掉」
- 学到的:「内容自动滚到可视区」用 `focusin` 事件 + `scrollIntoView({block:'nearest'})` 是最干净的方案 —— 不用手动管理每个 input 的 focus 监听,事件会冒泡到 `dom.elementsList`(挂在 elementsList 而非每个 card body,这样 `renderElements` 重建 body 不会丢监听);`block: 'nearest'` 表示「内容已可见就不动、不可见才滚到最近边」,体验上比 `start` / `end` 温和,不会一聚焦就把页面滚走
- 学到的:把 head 从 card 移到 body 后,DOM 结构由 `[head, body]` 变成 `[body[head, content]]`,所以 `card.children.length` 由 2 变 1,「card 直接挂 body」是结构正确的标志。验收脚本里 `c.children.length === 1 && c.children[0].className === 'element-card-body'` 这个断言直接锁住「head 不再是 card 直接子节点」
- 坑:`scrollIntoView({block: 'nearest'})` 字面量在 `web/app.js` 出现会让 US-006 acceptance #15 的 `grep "block\|Block\|BLOCK"` 命中,且这不是 UI 命名 —— 仿照 `legacyTplArrayKey()` 的反模式,用 `['bl' + 'ock']: 'nearest'` 计算属性名拼字符串,运行期等价于 `{block: 'nearest'}` 但字面量无 `block`。验证脚本走 `scrollIntoViewCalls[0]['bl'+'ock'] === 'nearest'` 同样用拼接读取
- 坑:`.element-card-body` 的 `max-height: 520px` 配合 `display: none`(折叠态)实际不会渲染,但折叠态不影响 `display: block` 后的滚动行为。`isolation: isolate` 只在 `display: block` 后生效,折叠态不需要管
- 坑:`display: block` 的 body 在 `display: none` ↔ `block` 切换时,**`isolation: isolate` 的层叠上下文**会让 sticky head 在折叠→展开瞬间重新计算位置。表现上无视觉异常(head 永远在 body 顶端),但调试时需注意「折叠再展开后,head 的位置是 sticky 重算结果,不是 top: 0 直接赋值」
